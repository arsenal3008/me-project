# AgentChecker — аудит безопасности и анализ кода

Дата: 2026-06-10 · Файл: `agent_checker.py` (1785 строк) · Автор обзора: Viktor

## TL;DR
Программа рабочая и функциональная, но есть **критичные проблемы безопасности** (отключённая проверка TLS, ключи в открытом виде и в логах) и **несколько реальных багов**, из-за которых часть функций не работает. Ниже — полный разбор и план модернизации.

---

## 🔴 Критично

### 1. Полностью отключена проверка TLS-сертификатов
```python
_ssl_ctx.check_hostname = False
_ssl_ctx.verify_mode = ssl.CERT_NONE
...
r = requests.post(url, json=body, headers=headers, timeout=timeout, verify=False)
```
- Соединения не проверяют сертификат → возможна **MITM-атака**: злоумышленник в сети может перехватить **API-ключи** (`Authorization: Bearer ...`), которые ты отправляешь.
- `InsecureRequestWarning` намеренно заглушается, чтобы это не было заметно.
- **Фикс:** включить проверку по умолчанию (`verify=True`, обычный SSL-контекст). Отключение оставить только как явную опцию «для localhost / самоподписанных сертификатов».

## 🟠 Высокий риск

### 2. API-ключи хранятся в открытом виде
- `providers.json` и `opencode.jsonc` пишутся как обычный JSON с ключами в plaintext, без ограничения прав доступа к файлу.
- **Фикс:** ставить права `0600` (только владелец) на `providers.json`; не дублировать ключи без необходимости; предупреждать пользователя.

### 3. Ключи попадают в логи (на диск)
```python
self._log('%s %s: %s%s' % (status, k[:48], msg, note))   # 48 символов ключа!
print('[%s] %s... -> HTTP 200' % (tag, k[:12]))
```
- Лог пишется в `logs/YYYY-MM-DD.log`. Туда утекают первые **48 символов** ключа (а это почти весь ключ у многих провайдеров).
- **Фикс:** маскировать ключи (`sk-…abcd`, показывать только хвост 4 символа).

### 4. Кнопка «Проверить» ключи провайдера не работает
```python
model_id = p['models'][0] if isinstance(p['models'][0], str) else ...
```
- `p['models']` — это **словарь**, а не список. `p['models'][0]` → `KeyError`. Функция падает всегда.
- Плюс там же `body = json.dumps({...})` передаётся в `http_post`, который кодирует JSON ещё раз → **двойная сериализация** (тело уходит как строка, а не объект).
- **Фикс:** `model_id = next(iter(p['models']))`, передавать dict, а не строку.

## 🟡 Средний риск

### 5. Сломано цветное логирование (опечатка)
```python
self.log.insert(tk.END, msg + '\n', tagS=tag)   # tagS — несуществующий аргумент
```
- Правильный вызов — позиционный тег: `self.log.insert(tk.END, msg + '\n', tag)`.

### 6. Жёстко зашитые пути конкретной машины
```python
r'C:\Users\msi\.config\opencode\opencode.jsonc'
r'C:\Users\msi\AppData\Local\Programs\Ollama\ollama.exe'
```
- Работает только на машине пользователя `msi`. На любой другой — не найдёт конфиг/Ollama.
- **Фикс:** `os.path.expanduser('~')`, `shutil.which('ollama')`, кроссплатформенно.

### 7. Баг в `_check_server_online`
- При отсутствии порта в URL по умолчанию берётся `11434` (порт Ollama) даже для https-провайдеров → проверка «онлайн» для удалённых API бессмысленна.

## 🟢 Низкий риск / качество кода

- **Подмена заголовков** `Origin/Referer: https://opencode.ai` и фейковый `User-Agent` для обхода Cloudflare — хрупко и может нарушать ToS провайдеров.
- **Наивный парсер JSONC** — не понимает блочные комментарии `/* */`, неточно обрабатывает кавычки.
- **Битый CSV** для непроверенных моделей: `'"%s","%s","%s",-.,\n'` — ломает структуру столбцов.
- **Мёртвый код**: `BUILTIN_MODELS = {}` и `_load_builtin()` ничего не делают, хотя в описании заявлен «встроенный каталог».
- **Дубль строки** `keys_by_prov[pn] = ...` дважды.
- Множество «глухих» `except Exception: pass`, проглатывающих ошибки.
- Один файл на 1785 строк, один огромный класс — сетевой код, хранение и UI перемешаны.
- Совместимость с Python 2 (EOL c 2020) тянет лишние shim-ы.
- Нет `requirements.txt`, типов, тестов.

## План модернизации (что делаю в PR)
1. **Безопасность:** TLS-проверка включена по умолчанию (+ явная опция для localhost); маскирование ключей в логах; права `0600` на `providers.json`.
2. **Исправление багов:** логирование (`tagS`→tag), кнопка проверки ключей (dict + dict-body), кроссплатформенные пути, CSV, порт-дефолт.
3. **Современный UI с вкладками** (`ttk.Notebook`): «Дашборд», «Модели», «Провайдеры и ключи», «Лог», «Настройки» — аккуратная тёмная/светлая тема, иконки, карточки статистики.
4. **Чистка кода:** убран Python 2-код, добавлены docstrings и аккуратная структура, удалён мёртвый код.
5. **`requirements.txt`** (закрывает Issue #3).

> ⚠️ GUI я не могу запустить в своём окружении (нет дисплея), поэтому проверяю синтаксис и логику, а финальный визуальный тест — за тобой локально. Все изменения придут отдельным PR, ничего не ломая в `main` до твоего ревью.
